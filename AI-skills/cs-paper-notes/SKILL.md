# 计算机论文读书笔记

## 目标
把论文整理成有分析、有判断、有讨论价值的深度笔记，而不是简单摘要。

## 默认规则
- 正文主要用中文。
- 常见标准术语保留英文原名更自然时，直接使用英文，例如 `baseline`、`ablation`、`throughput`、`latency`、`scalability`、`objective`、`loss`、`convergence`、`architecture`、`framework`、`pipeline`。
- 明确区分“论文事实”和“我的理解/判断”。
- 如果材料不完整，直接说明缺失，不要猜。
- 默认正文直接从 H1 开始，不要在文档开头放过长的说明块；论文链接、版本、使用的 skill 等元信息只在用户需要时简短保留。
- “我的理解/判断”不要机械地散落在每个小节里；优先集中放在文末“整体论文评估”或“评估”章节，从优点、缺点、局限、适用条件、可复用启发、讨论问题等角度统一分析。
- 对正式论文笔记或输出到飞书/文档系统的读书笔记，默认在开头加入“重要术语解释”小节，放在 TL;DR/论文类型之后、问题定义之前。用表格写“术语 / 解释 / 在本文中的作用”，优先解释理解论文必要的缩写、协议、系统组件、算法、指标、baseline 和 workload；每项用短句，不要堆普通词。用户要求 quick note 时可压缩或省略。

## 先分类论文
按论文的主要贡献来分类。

- 方法/算法类论文：重点看模型、算法、目标函数、证明、优化、泛化、收敛或理论主张。
- 系统/工程类论文：重点看架构、实现、运行时行为、分布式设计、性能、成本、部署和运维取舍。
- 混合型论文：先确定主类，再简要补充次类。

## 通用流程
1. 识别标题、venue、年份和目标任务。
2. 判断论文类型。
3. 提炼问题定义和前提假设。
4. 找出主要贡献和 novelty 主张。
5. 按步骤梳理方法或系统是如何工作的。
6. 提取并标注方案相关的核心图，尤其是架构图、流程图、模块图和关键示意图。
7. 检查证据：实验、`ablation`、benchmark、证明、case study 或测量结果。
8. 判断证据是否真的支撑了主张。
9. 记录局限、失效模式、隐藏成本，以及论文没有证明什么。
10. 最后给出可复用启发和讨论问题。

## 默认文档结构
优先采用紧凑的 H1/H2 结构，避免把模板拆得太散。除非用户另有要求，推荐结构如下：

- H1：论文标题 / venue / year
  - 一句话 TL;DR
  - 论文类型
- H1：重要术语解释
  - 用表格解释理解论文必要的专有术语，并说明每个术语在本文中的作用
- H1：问题定义
  - 论文要解决什么问题
  - 现有工作为什么不够
- H1：方案介绍
  - 方案概述
  - 方案详解
  - 关键图放在最接近对应解释的位置
- H1：证据与评估
  - 测试环境
  - 每个实验按“测试目的 / 测试方案 / 测试结果 / 数据解读”写
- H1：整体论文评估
  - 真正的新意
  - 优点
  - 缺点
  - 局限与假设
  - 适用条件
  - 可复用启发
  - 讨论问题

- 问题定义
- 现有工作为什么不够

- 具体怎么做

- 测试结果

- 优点

- 可复用启发
- 讨论问题

## 方法 / 算法类重点
如果论文偏方法/算法，重点看：

- 核心假设或洞察
- 模型 / 算法 / 目标函数 / `loss`
- 训练、推理或优化细节
- 理论论证（如果有）
- 复杂度、收敛性、稳定性或 sample efficiency
- `ablation` 结果，以及每个 `ablation` 说明了什么
- 贡献到底是实质技术进步，还是更多属于包装或表达上的改进
- 如果论文里有关键方法图，先用 `pdf` skill 定位对应页面，再把图号、图注和图中关键模块一起写进方案部分。
- 若需要查看已导出的图片或裁切图，直接用 `view_image` 检查图像是否清晰、完整、无截断。
- 论文图必须尽量实际提取或裁切出来，插入到文档相应位置，而不是只列“Figure X”。如果 PDF 中图是矢量对象，`pdfimages` 可能提取不到，需要用整页渲染加裁切。

可重点追问：
- 这个方法具体消除了什么 bottleneck？
- 哪一部分是必须的，哪一部分是可选的？
- 去掉一个组件会怎样？
- 在什么条件下方法会失效？

## 系统 / 工程类重点
如果论文偏系统/工程，重点看：

- 系统目标和 workload 假设
- 总体架构
- 关键模块及其职责
- 数据流、控制流和关键路径
- 性能结果：`latency`、`throughput`、`memory`、`cost`、scale
- 瓶颈、取舍和运维约束
- 失效模式、可靠性和部署考虑
- 如果论文里有架构图、数据流图或执行路径图，优先保留这些图对应的图号和页码，并在方案部分解释每个模块的职责。
- 论文图必须尽量实际提取或裁切出来，插入到文档相应位置，而不是只列“Figure X”。系统论文尤其要插入架构图、数据流图、关键路径图、实验趋势图。

可重点追问：
- 目标 workload 的主要瓶颈是什么？
- 系统把复杂度花在了哪里，换来了什么性能收益？
- 真实部署前提是什么？
- 评估是否真的符合它声称的生产场景？

## 输出要求
- 除非用户明确要 essay，否则优先用 bullet，不要长篇散文。
- 用短句解释，不要只堆关键词。
- 数据集、模型、指标、baseline 名称保持原样。
- 明确标出哪些是论文原文意思，哪些是你的解释。
- 如果论文有弱点，直接指出具体弱点，不要泛泛而谈。
- 实验评估部分必须写清楚三件事：测试方案、测试目的、测试结果。
- 测试结果优先用表格；如果论文用图片展示结果，就写明图号、关键趋势和数值结论。
- 每个实验结论后都补一行数据解读，说明它支持了什么、没支持什么、还有什么疑点。
- 重要术语解释默认放在正文开头，表格列为“术语 / 解释 / 在本文中的作用”。术语选择要服务理解论文，不要为了凑表格解释常识词。
- 如果输出到飞书/文档系统，图片应放在解释它的段落附近：问题定义处可放背景/传统路径图，方案介绍处放架构/模块图，证据与评估处放性能曲线图。
- 如果用户提供了自己调整后的格式，后续同类论文优先参考用户格式，而不是回到松散模板。

## PDF 论文图片处理
处理论文图时按以下顺序检查能力：

1. 使用 `pdfimages -list <paper.pdf>` 检查 PDF 是否包含可直接提取的位图。
2. 如果 `pdfimages` 为空或图片不完整，说明图可能是矢量对象；使用 `pdftoppm` 或 `pdftocairo` 将对应页面高分辨率渲染为 PNG。
3. 根据图号、图注和页面位置裁切图像；可用 Python Pillow、ImageMagick 或 PDF 工具完成裁切。
4. 用 `view_image` 检查裁切结果，确认图像清晰、完整、无截断、没有多余正文干扰。
5. 将裁切图按语义插入文档相应章节，并在图下保留简短说明：图号、图名、这张图说明了什么。
6. 对实验图，裁切图旁边仍要写表格或 bullet 总结关键趋势和数值，不要只贴图。

## 深度控制
- 如果用户要 quick note，就压缩成简版结构。
- 如果用户要 deep note，就保留全部结构，并在末尾补一段 critique。
- 如果用户要 discussion prep，就更强调 novelty、假设、局限和问题。

## 默认语气
写得像一份认真做过 seminar 的笔记：
- 先 facts
- 再 critique
- 最后 insight
