# 计算机技术教程写作

使用本技能来编写或改进计算机技术教程。目标不是简单罗列概念，而是把系统机制讲清楚，让读者知道：

> 系统在算什么？状态在哪里？什么被切分？什么被复制？什么需要移动？在哪里同步或合并？

## 核心教学原则

复杂技术主题不要一开始就给一张大而全的总览图。应按理解路径逐层展开：

1. 先讲最小可运行场景。
2. 说明这个场景中有哪些数据、状态和计算步骤。
3. 解释为什么单机、单卡或简单方案会遇到瓶颈。
4. 一次只引入一种扩展或并行策略。
5. 对每种策略明确回答：切什么、复制什么、移动什么、在哪里同步。
6. 最后再给总结表或总览图。

## 推荐教程结构

除非用户明确要求其他结构，优先使用下面的结构。

### 1. 问题背景

说明这个技术要解决什么问题：

- 为什么需要这个技术？
- 简单方案是什么？
- 简单方案在哪些条件下不够用？

### 2. 基本计算结构

先讲单机、单进程、单 GPU 或最小系统版本。

要说明：

- 输入是什么？
- 输出是什么？
- 中间状态是什么？
- 核心计算路径是什么？
- 关键数据结构或状态有哪些？

可以使用形状、ID、rank、partition、伪代码或简单例子帮助理解。

### 3. 单机 / 单卡 / 基线执行

在进入分布式或并行之前，必须讲清楚基线场景：

- 所有状态在哪里？
- 是否需要跨节点、跨 GPU、跨进程通信？
- 是否需要 CCL、RPC、锁、共识协议、同步机制？
- 当前瓶颈是什么？

示例：

```text
单 GPU 训练：
    参数、activation、gradient、optimizer state 都在同一张 GPU 上
    forward / backward / optimizer 都只读写本地 HBM
    没有跨 GPU 数据交换
    因此不需要 CCL
```

### 4. 扩展或并行策略

逐个介绍扩展策略。不要一开始把所有策略塞进一张复杂图里。

每种策略都要回答：

```text
切分对象：
复制对象：
本地计算：
移动的数据：
合并 / 同步方式：
同步发生时间：
主要收益：
主要代价：
```

### 5. 关键机制细节

对最重要的策略，补充更细的时序图或流程图。

要把同步点放到时间线上，例如：

```text
计算前
计算中
计算后
forward 前
backward 中
backward 后
commit 前
cache miss 时
leader election 时
recovery 时
```

不要只写“然后同步”，要说明同步的对象和时机。

### 6. 对比总结

在读者理解各个机制之后，再给对比表。对比维度可以包括：

- 切分对象
- 通信对象
- 同步时机
- 优点
- 代价
- 适用场景
- 常见误解

### 7. 实现映射

把概念映射到真实系统或框架。可以分层说明：

```text
用户代码 / API
框架运行时
通信库 / 协议层
内核 / 驱动 / runtime
硬件 / 网络拓扑
```

这样能帮助读者理解：为什么一行简单 API 背后会触发复杂系统行为。

### 8. 性能或问题诊断

如果主题涉及工程实践，要给读者一个排查思路：

- 瓶颈在计算、内存、网络还是调度？
- 是否有同步等待？
- 是否有数据倾斜？
- 是否有热点？
- 是否能与计算重叠？
- 是否受拓扑、batch size、message size 影响？

### 9. 推荐学习顺序

最后给读者一个学习路径：

1. 先理解单机计算结构。
2. 再理解为什么需要扩展。
3. 再逐个学习并行或分布式策略。
4. 最后学习底层实现和性能优化。

## 写作规则

### 先讲具体计算，再讲抽象概念

优先使用这样的路径：

```text
input -> compute -> intermediate state -> output -> update
```

然后解释每个对象是什么。不要一开始就上分类表、术语表或复杂架构图。

### 单机和分布式必须分开讲

很多教程讲不清楚，是因为一开始就把单机逻辑和分布式同步混在一起。

应明确说明：

```text
单机/单卡：
    所有状态本地保存
    不需要跨设备同步

多机/多卡：
    某些对象被切分或复制
    因此需要通信、同步或合并
```

### 一张图只讲一个核心概念

初次讲解时，优先使用多张简单图，而不是一张大而全的图。

推荐图示顺序：

1. 单机/单卡基线图
2. 策略 A 的切分图
3. 策略 B 的切分图
4. 策略 C 的切分图
5. 策略 A 的通信细节图
6. 策略 B 的通信细节图
7. 最终对比表

### 明确说明状态在哪里

系统教程必须讲清楚 state ownership。

常见状态包括：

- 配置状态
- 元数据
- 输入数据
- 中间结果
- 缓存
- 参数
- 梯度
- 日志
- checkpoint
- shard
- replica
- ownership
- lease
- lock

很多误解来自读者不知道“这个东西到底在哪台机器、哪张 GPU、哪个进程里”。

### 明确区分切分、复制、移动、合并

每个策略都要尽量使用精确动词：

- 切分：shard / partition / split
- 复制：replicate / copy
- 移动：send / route / transfer
- 合并：reduce / gather / merge / commit
- 同步：sync / barrier / consensus / allreduce

不要笼统说“通信数据”，要说清楚通信的到底是什么：

```text
gradient
parameter shard
activation
token activation
log entry
metadata
cache line
page
partial result
index entry
```

### 表格用于复习，不用于首次解释

表格很适合总结，但不适合第一次解释复杂机制。

推荐顺序：

```text
直觉解释 -> 图示 -> 细节机制 -> 对比表
```

## 图示原则

当主题涉及位置、归属、流动、层级、同步时，应主动使用图。

| 需求 | 推荐图 |
| --- | --- |
| 数据流 / 执行流 | flowchart |
| 同步时机 | sequence / timeline |
| 分片与归属 | partition diagram |
| 软件栈 | vertical stack |
| 网络或硬件结构 | topology diagram |
| 复习对比 | table |

图里的每个框和箭头都要尽量回答：

- 这个框是什么？
- 它在哪里？
- 它持有什么状态？
- 箭头传的是什么？
- 箭头发生在什么时候？
- 这个操作是计算、通信还是同步？

## 深度控制

根据用户目标调整深度。

### 用户说“入门 / 帮我学习”

- 先讲直觉。
- 多用具体例子。
- 少放实现细节和边界情况。
- 每节只引入少量新概念。

### 用户说“深入原理”

- 补充时序、数据结构、状态机、通信路径。
- 说明关键 trade-off。
- 讲清楚实现变体。

### 用户说“面试 / 复习”

- 加总结表。
- 加常见误区。
- 加对比问题。
- 加一句话记忆法。

### 用户说“工程落地”

- 增加指标、日志、配置项、调参方向。
- 增加故障模式。
- 增加排查 checklist。
- 说明真实框架或系统里的实现差异。

## 审阅 Checklist

完成教程前检查：

1. 是否先讲了基本计算结构？
2. 是否讲清楚单机/单卡场景？
3. 是否避免把分布式通信误写成单机也需要？
4. 每种策略是否说清楚切分对象？
5. 是否说清楚什么东西被复制？
6. 是否说清楚什么数据会移动？
7. 是否说清楚同步或合并发生在什么时候？
8. 是否避免一开始使用过于复杂的大图？
9. 是否先解释概念，再用表格总结？
10. 是否给了读者继续学习或排查问题的路径？

## 常见修改方式

如果一篇教程读起来很乱，优先这样改：

- 把单机/单卡基线提前。
- 把一张大图拆成多张小图。
- 在每个策略前补一句“切分对象是什么”。
- 把“通信数据”改成具体对象名。
- 给同步点加时间线。
- 把概念层和实现层分开。
- 增加一个具体例子，例如 4 张 GPU、3 个 shard、2 个 replica。
- 把总结表放到后面，而不是开头。

## 推荐表达方式

优先使用清楚、克制、准确的技术表达。

推荐：

```text
可以把它理解成……
关键点是……
这里同步的是 gradient，不是 input batch。
单卡时不需要 CCL，因为没有跨 GPU 数据交换。
具体实现可能因框架不同而变化，但核心思想是……
```

避免：

```text
显然……
众所周知……
简单来说就是通信一下……
所有实现都是这样……
```

如果实现因框架而异，应明确说明：

```text
在 PyTorch DDP 中，常见做法是……
在 Megatron 风格实现中，通常会……
具体实现可能不同，但抽象机制是……
```
